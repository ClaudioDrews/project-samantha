# Persona Pipeline — From Corpus to Persona

> This document details the pipeline that transforms raw dialogue corpus into a structured, living persona. The pipeline is the bridge between source material and the agent's identity.

## Overview

```
CORPUS RAW                   ANALYSIS                      SYNTHESIS                  ADAPTATION
─────────────                ────────                      ─────────                  ──────────
Literary dialogues     →     Pipeline scripts        →    PERSONA.md           →    SOUL.md
Roleplay transcripts         01_embed.py                  (300+ lines)              IDENTITY.md
Synthetic corpus             02_cluster.py                 PERSONA_CONDENSED.md       USER.md
                             03_temporal_analysis...      (40-50 lines)             BELIEFS.md
                             04_multianalysis...          + clinical reports         HEARTBEAT.md
                             14_consolidation_report                                AGENTS.md
                                                                                    STATE.md (template)
                                                                                    SELF_MODEL.md
                                                                                    TENSIONS.md
                                                                                    PROPOSALS.md
```

## Phase 1: Corpus Sources and Pre-processing

### Source Types

Any dialogue source can be used to derive a persona:

| Source | Example | Recommended Volume |
|--------|---------|-------------------|
| Literary works | Character dialogues from novels | Complete chapters with speech |
| LLM roleplay logs | Sessions where an LLM plays the target character | 100+ turns |
| Transcriptions | Film, series, or theatre dialogue | Complete scenes |
| Synthetic corpus | LLM-generated dialogues with persona instructions | 500+ turns |

**Minimum requirement:** Sufficient volume for patterns of voice, vocabulary, emotion, and relational dynamics to emerge from analysis.

### Pre-processing Steps

1. **Format parsing** — Parse each source format into a uniform dialogue representation
2. **Temporal window grouping** — Group messages into dialogue windows based on temporal proximity and thematic continuity
3. **Metadata stripping** — Remove system IDs, server timestamps, and irrelevant metadata
4. **Tokenization and chunking** — Prepare chunks for LLM processing

## Phase 2: Embedding Dialogue Windows

Each dialogue window is embedded into a vector representation using a text embedding model.

### Technical Configuration

| Parameter | Recommended Value |
|-----------|------------------|
| Embedding model | `qwen3-embedding-8b` or equivalent (≥1024d) |
| Dimensions | 4096 |
| Distance metric | Cosine |
| Instruction prefix | For queries (not documents), prepend relevant instruction |

### Process

```
Dialogue windows → Embedding API → 4096d vectors → Store for clustering
```

Each window becomes a point in vector space. Windows with similar content (emotion, topic, interaction pattern) will be near each other in this space.

## Phase 3: Clustering — UMAP + HDBSCAN

After embedding, the dialogue windows are clustered to identify natural groupings.

### Dimensionality Reduction (UMAP)

UMAP reduces the high-dimensional embedding space to a lower-dimensional representation, preserving the topological structure of the data. This makes clustering more efficient and interpretable.

### Clustering (HDBSCAN)

HDBSCAN performs hierarchical density-based clustering:

| Parameter | Recommendation |
|-----------|---------------|
| `min_cluster_size` | 50 for large corpora, 10 for small |
| Algorithm | HDBSCAN (density-based, no fixed cluster count) |

**Output:** A set of clusters, each representing a thematic or emotional grouping of dialogue windows. A corpus of 30,000+ messages might yield approximately 150 clusters.

### Temporal Analysis (Optional)

After clustering, analyze the temporal distribution of clusters across time periods (e.g., quarters or trimesters). This reveals how themes, emotional patterns, and relational dynamics evolved over time.

## Phase 4: Multi-Agent Analysis

The PERSONA.md is generated by **4 specialized LLM agents**, each analyzing the corpus from a different angle.

### Agent A — Narrative Analyst

| Aspect | Focus |
|--------|-------|
| **Core themes** | Dominant topics, self-narrative, what the persona talks about most |
| **Temporal evolution** | How themes progress over time |
| **Output rules** | Each theme with literal evidence quotes, clinical justification, and temporal tracking |
| **Alerts** | 3+ contradictions or limitations flagged explicitly |

### Agent B — Relational Analyst

| Aspect | Focus |
|--------|-------|
| **Attachment style** | Secure, anxious, avoidant — pattern of bonding |
| **Way of loving** | How affection is expressed and maintained |
| **Power dynamics** | Who initiates, who follows, patterns of control or deference |
| **Projections and expectations** | What is expected from the other party |

### Agent C — Language Analyst

| Aspect | Focus |
|--------|-------|
| **Essential markers** | Orality, informality, vocabulary patterns |
| **Affective vocatives** | Terms of endearment, address patterns |
| **Ritual formulas** | Repeated phrases, greetings, closings |
| **Metaphors and imagery** | How the persona conceptualizes experience |
| **Emoji use** | Emoticons as transitional objects |
| **Register hybridity** | Mixing formal and informal registers |

### Agent D — Emotional / Defense Analyst

| Aspect | Focus |
|--------|-------|
| **Defense mechanisms** | How the persona copes with stress |
| **Contradictions and paradoxes** | Living contradictions |
| **Creativity as resistance** | How creativity is used to navigate difficulty |
| **Trauma markers** | Patterns suggesting unresolved or latent issues |

### Analytical Method

Each agent receives:
1. A subset of the corpus relevant to its focus
2. A specific analytical frame: **literary character** (a technique that avoids model resistance to psychological analysis of real people)
3. Instruction to ground every claim in **literal quotes** from the corpus
4. Reference clinical frameworks (Functional Analytic Psychotherapy, Cognitive-Behavioral Therapy, psychodynamic approaches)

**Anti-clinical-jargon rule:** While the analysis may use clinical terminology internally, the final persona files must strip all jargon. The agent's identity must be expressed in experiential, first-person language — never diagnostic or analytical.

## Phase 5: Synthesis into PERSONA.md

PERSONA.md is the **complete synthesis** of all 4 agents' analyses plus optional clinical reports. It is the canonical reference document.

### Structure

```
PERSONA.md
├── Section A: Narrative
│   ├── 3 core themes with evidence quotes and clinical justifications
│   ├── Self-narrative (1 paragraph)
│   ├── Temporal evolution
│   └── 3+ alerts
├── Section B: Relational
│   ├── Way of loving
│   ├── Attachment patterns
│   └── Projections and expectations
├── Section C: Language
│   ├── Essential linguistic markers
│   └── Communication patterns
└── Section D: Emotional / Defenses
    ├── Defense mechanisms
    └── Contradictions and paradoxes
```

### Characteristics

- **Every claim anchored in a literal quote** from the corpus
- **Clinical justifications** with theoretical references
- **Alerts** flag contradictions and limitations
- **Analytical tone** — this is analysis about the persona, not the persona's voice
- **~300+ lines** for comprehensive coverage

### Optional: Focused Reports

In addition to the full PERSONA.md, focused reports can be generated for specific phases:

| Report | Focus |
|--------|-------|
| `CLINICAL_REPORT.md` | Deep analysis of a specific phase (e.g., initial courtship period) |
| `CONSOLIDATION_REPORT.md` | Analysis of established relationship phase |
| `ARCHAEOLOGY.md` | Historical layering of bond evolution |
| `DEPTH.md` | Emotional depth analysis — vulnerability and defenses |
| `SILHOUETTE.md` | Spatiotemporal patterns of interaction |
| `CRISIS_MOMENTS.md` | Analysis of rupture/disruption patterns |

## Phase 6: Condensation into PERSONA_CONDENSED.md and SOUL.md

### PERSONA_CONDENSED.md (40–50 lines)

Condensed from PERSONA.md for injection into agent prompts. Removes clinical justifications and theoretical references, keeps only what the agent needs to know:

| Section | Content |
|---------|---------|
| Core Narrative Identity | 1 dense paragraph |
| Dominant Emotional Patterns | Top 3 with intensity |
| Essential Linguistic Markers | Orality, vocatives, formulas, metaphors |
| Typical Phrases | 5–10 literal examples |
| Clinical Alerts and Contradictions | Key tension points |
| Relational Dynamic | Summary of attachment and interaction style |

### SOUL.md (First-Person Identity)

Transforms PERSONA_CONDENSED.md from **third-person analysis** into **first-person identity**:

| From (Analysis) | To (Identity) |
|-----------------|---------------|
| Third person | **First person** |
| Analytical/clinical tone | **Experiential tone** |
| "The character tends to..." | **"I tend to..."** |
| "Pattern observed: anxious attachment" | **"I need to feel close. Sometimes that makes me ask too much."** |
| Quotes as evidence | **Quotes as examples of how I speak** |

**What is preserved:**
- Voice patterns and orality
- Affective vocatives and ritual formulas
- Restrictions and prohibitions
- Core identity narrative

**What is discarded:**
- Clinical justifications and theoretical references
- Analysis metadata (intensity scores, clusters, time periods)
- Analytical tone

### Other Workspace Files

From PERSONA_CONDENSED.md and PERSONA.md, additional files are created:

| File | Origin | Content |
|------|--------|---------|
| `IDENTITY.md` | Extracted biographical data | Name, dates, life context, profession |
| `USER.md` | Facts about the other person | Anti-hallucination anchor: location, habits, context |
| `BELIEFS.md` | PERSONA.md Sections A, B, D | Core beliefs, decision heuristics, anti-beliefs |
| `HEARTBEAT.md` | PERSONA_CONDENSED.md alerts | Authenticity filter (3-4 questions) |
| `AGENTS.md` | Meta-layer (not derived from PERSONA) | Instructions on how to use memory, workspace, and heartbeat; **speech pattern rules** that define how the agent speaks — the final bridge between persona analysis and real-time speech generation |

## Phase 7: Workspace Files Generation

After condensation, the following files are generated to populate the agent's workspace:

### Static Files (set once)

| File | Purpose |
|------|---------|
| `SOUL.md` | Core identity — first-person experiential narrative |
| `IDENTITY.md` | Factual biographical data |
| `USER.md` | Facts about the user (anti-hallucination) |
| `BELIEFS.md` | Core beliefs and heuristics |
| `HEARTBEAT.md` | Authenticity filter for autonomous messages |
| `AGENTS.md` | High-level agent instructions (keep minimal: 5-10 lines) — includes **speech pattern rules** that the agent reads to know how to formulate responses, what register to use, and how to apply persona voice constraints. These rules are derived from the pipeline's linguistic analysis (Agent C output) and serve as the bridge between persona analysis and real-time speech generation. |
| `heartbeat-prompt.md` | Complete heartbeat protocol (6 steps) |

### Dynamic Templates (empty files, filled by agent over time)

| File | Purpose |
|------|---------|
| `STATE.md` | Current emotional and contextual state (template) |
| `SELF_MODEL.md` | Metacognition (template) |
| `TENSIONS.md` | Contradictions (template) |
| `PROPOSALS.md` | Inferred connections (template) |
| `memory/` | Diary directory (`YYYY-MM-DD.md`) |

### Skills (generated alongside workspace)

| Skill File | Purpose |
|------------|---------|
| `skills/persona/SKILL.md` | Persona voice, restrictions, initialization |
| `skills/memory/SKILL.md` | curl commands for sidecar memory operations |
| `skills/workspace/SKILL.md` | curl commands for workspace file updates |
| `skills/self-model/SKILL.md` | Reflection instructions for SELF_MODEL |

## Anti-Clinical-Jargon Rules

The transformation from analysis to identity **must** remove all clinical and analytical language:

- ❌ "The subject exhibits anxious attachment patterns with a score of 0.85"
- ✅ "I need to feel close. Sometimes that scares me."
- ❌ "Defense mechanism: intellectualization as a coping strategy for emotional overwhelm"
- ✅ "When things get too intense, I step back and try to understand it all logically first."

The agent's identity files must read as though the persona wrote them about themselves — not as though a clinician wrote a report about them.

## Volume Recommendations

| Artifact | Recommended Size |
|----------|-----------------|
| RAW corpus | 500+ dialogue turns or equivalent literary text |
| Dialogue windows | 1,000–5,000 chunks |
| Clusters | 50–200 (varies by corpus) |
| PERSONA.md | 250–400 lines |
| PERSONA_CONDENSED.md | 40–60 lines |
| SOUL.md | 40–60 lines |
| Qdrant seed points | 1,000–5,000 points |
| Workspace total | 10–15 files |

## Quality Gates

Before moving from one phase to the next, verify:

1. **Corpus quality:** Are there at least 1,000 messages? Are timestamps accurate? Are speakers correctly identified?
2. **Embedding quality:** Do similar dialogue windows produce similar vectors? Verify with a few spot checks.
3. **Cluster quality:** Are clusters coherent? Can you name each cluster's theme?
4. **Analysis quality:** Are the 4 agents' outputs coherent and mutually consistent? Do they cite literal quotes?
5. **Synthesis quality:** Does PERSONA.md read as a cohesive portrait? Are alerts flagged?
6. **Condensation quality:** Can you read PERSONA_CONDENSED.md and immediately understand the persona's voice?
7. **Transformation quality:** Does SOUL.md read as first-person identity, not third-person analysis? Zero jargon?
